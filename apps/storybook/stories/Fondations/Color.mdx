# ShelterUI Design System colors

These palettes are intended to be the foundation of “ShelterUI”, enabling the creation of harmonious and coherent light and dark themes.
They ensure a smooth, pleasant user experience, while offering flexibility for future adjustments.

## Color Palette

The “ShelterUI” color palette is designed to form the basis of the design system. Each color has been chosen to offer excellent legibility and visual harmony.
The palette includes variations of each color, from the lightest to the darkest, enabling the creation of consistent light and dark themes.\
Colors are named by their hue and intensity, e.g. color-blue-500.\
This palette is intended for generating ShelterUI themes and should not be used directly.

## Primitive Colors Usage

### **What are Primitive Colors?**

Primitive colors in ShelterUI are the foundational building blocks of the design system's color palette. These colors provide the full range of shades and variations for each hue (e.g., Blue, Purple, Red). They are defined systematically using SCSS variables to ensure consistency and modularity across the entire system.

### **Rules for Using Primitive Colors**

**Primitive colors should never be used directly in components or styles.** Their purpose is to act as a foundational layer upon which theme tokens are built. These theme tokens encapsulate the contextual meaning and usage of colors (e.g., primary, secondary, background) within specific themes like light or dark.

Using primitive colors directly can lead to:

- Inconsistent application of colors across the design system.
- Lack of alignment with thematic contexts (e.g., light vs. dark modes).
- Difficulty in maintaining and evolving the design system.

Instead, all styling must refer to **theme tokens** defined in the respective themes (e.g., `light-theme`, `dark-theme`).

### **Accessing Primitive Colors**

Primitive colors are available **exclusively** as SCSS variables, adhering to the following naming convention:

```
color-{hue}-{variant}
```

#### **Example:**

For the Blue hue with a `500` intensity, the SCSS variable is named:

```scss
$color-blue-500
```

You can access the variables in your SCSS files as follows:

```scss
@use 'variables/color';

.my-component {
  background-color: #{color.$color-blue-500}; // Primitive color (not recommended for direct use)
}
```

---

### **Why Use Theme Tokens Instead of Primitives?**

1. **Encapsulation of Context:**
   Theme tokens encapsulate the meaning of colors based on their context. For example:

- `--color-primary-main-light` refers to the primary color in a light theme.
- `--color-primary-main-on` refers to the text color that contrasts with the primary background.

2. **Adaptability Across Themes:**
   Theme tokens ensure that the correct colors are applied for light, dark, or future themes (e.g., high contrast).

3. **Improved Maintainability:**
   With primitives abstracted into tokens, changing a color or adjusting a theme becomes a single update that cascades throughout the system, reducing redundancy and errors.

---

### **Best Practices**

- **Do not directly reference primitives:** Always use theme tokens to ensure consistency and proper contextual usage.
- **Follow the naming convention:** Use the `{hue}-{variant}` pattern for primitives (e.g., `color-blue-500`), but rely on theme-defined tokens for styling components.
- **Leverage the design tokens:** Theme tokens like `--color-primary-main-light` are optimized for both themes and accessibility standards.

By following these rules, ShelterUI ensures a modular, scalable, and consistent approach to color usage across all components and themes.
